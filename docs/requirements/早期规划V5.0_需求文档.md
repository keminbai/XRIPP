);

IF EXISTS (SELECT 1 FROM tender_download_logs WHERE user_id=@userId AND tender_id=@tenderId)
    -- 不扣费，直接返回下载链接
ELSE
    -- 扣1次额度，插入记录

### 1.2 合伙人福利分配机制（🆕 新增）

**业务规则**：
- 合伙人拥有**福利池**（由总部授予），可分配给名下会员
- 福利类型包括：标书下载次数、收费活动免费名额、报告下载次数、供应商入驻名额等
- 分配后，合伙人自身额度扣减，会员额度增加

**流程图**：
```text
[总部] 给上海合伙人授予福利池
    ├─ 标书下载次数：100次
    ├─ 收费活动名额：10个
    └─ 供应商入驻名额：5个
        ↓[上海合伙人] 登录后台 → 进入"福利发放"模块
    ↓
[选择会员] 搜索手机号/公司名 → 选择张三（VIP会员）
    ↓
[分配福利] 
    ├─ 额外赠送标书下载10次
    ├─ 赠送收费活动名额1个
    └─ 点击"确认发放"
        ↓
[系统执行]
    ├─ 合伙人福利池：标书次数 100→90、活动名额 10→9
    ├─ 张三权益：标书次数 50→60、活动名额 0→1
    └─ 记录日志（谁给谁发了什么，时间）
```

**数据库设计**：
```sql
-- 合伙人福利池
CREATE TABLE partner_benefit_pool (
    id BIGINT IDENTITY(1,1) PRIMARY KEY,
    partner_id BIGINT NOT NULL,
    benefit_type VARCHAR(50) NOT NULL, -- tender_download/activity_free/report_download/supplier_quota
    total_quota INT NOT NULL,
    used_quota INT DEFAULT 0,
    remaining_quota AS (total_quota - used_quota)
);

-- 福利分配记录
CREATE TABLE benefit_grant_logs (
    id BIGINT IDENTITY(1,1) PRIMARY KEY,
    partner_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    benefit_type VARCHAR(50) NOT NULL,
    grant_amount INT NOT NULL,
    grant_time DATETIME DEFAULT GETDATE(),
    operator_id BIGINT NOT NULL
);
```

### 1.3 活动举办完整流程（🆕 新增）

**步骤1：活动发布**
```text
[发起人] 总部/合伙人 → 登录后台 → "信息发布系统" → "活动发布"
    ↓
[填写信息]
    ├─ 活动类型：亨嘉之会/公益行/出海考察/行业沙龙
    ├─ 活动标题、时间、地点
    ├─ 活动费用：免费 / 199元 / 299元 / 自定义
    ├─ 报名人数限制：50人
    ├─ 是否主页展示：是/否
    ├─ 发布城市：全国/上海/北京
    └─ 上传宣传图、活动议程
        ↓
[提交审核]
```

**步骤2：审核流**
```text
[审管1] 收到通知 → 查看内容 → 批准/驳回
    ↓ (批准)
[审管2] 收到通知 → 查看内容 → 批准/驳回
    ↓ (批准)
[系统] 自动生成活动编号（如：总250108A010022）
    ↓
[前台] 活动中心显示该活动
```

**步骤3：会员报名**
```text
[会员] 前台看到活动 → 点击"立即报名"
    ↓
[判断费用]
    ├─ 免费活动 → 直接报名成功
    └─ 收费活动 → 检查权益
        ├─ 有免费名额 → 扣减1次名额 → 报名成功
        └─ 无名额 → 显示支付二维码 → 扫码支付 → 报名成功
            ↓[系统] 记录到 activity_registrations 表
```

**步骤4：活动记录（线下举办后）**
```text
[发起人] 登录后台 → "活动管理" → "活动记录"
    ↓
[录入信息]
    ├─ 实际参与人数：45人
    ├─ 现场照片上传
    ├─ 活动总结（200字）
    └─ 标记状态为"已完成"
        ↓
[导出] 参会企业名单Excel（含公司名、联系人、电话）
```

### 1.4 续费机制（🆕 新增）

**合伙人续费**（2-3年一次）
```sql
-- 合伙人表增加字段
ALTER TABLE partners ADD 
    join_date DATE NOT NULL,
    expire_date DATE NOT NULL,
    renewal_reminder_sent BIT DEFAULT 0;

-- 定时任务：到期前30天发送提醒
每天凌晨2点执行：
    SELECT * FROM partners 
    WHERE expire_date <= DATEADD(DAY, 30, GETDATE())
    AND renewal_reminder_sent = 0;
    
    → 发送短信/邮件："您的合伙人资格将于XX到期，请及时续费"
    → 更新 renewal_reminder_sent = 1
```

**会员续费**（每年）
```sql
-- 会员表已有字段
member_expire_time DATETIME

-- 定时任务：到期前7天发送提醒
每天凌晨3点执行：
    SELECT * FROM users 
    WHERE member_level IN ('VIP','SVIP')
    AND member_expire_time <= DATEADD(DAY, 7, GETDATE());
    
    → 发送站内信 + 短信："您的VIP会员将于XX到期"

-- 到期后自动降级
每天凌晨4点执行：
    UPDATE users 
    SET member_level = 'FREE', view_quota = 0
    WHERE member_expire_time < GETDATE();
```

## 第四部分：技术架构与实施方案

### 4.1 技术栈（最终确认）

| 层级 | 技术 | 版本 | 理由 |
|------|------|------|------|
| **前端-PC门户** | Nuxt.js | 3.10+ | SSR必须，SEO友好 |
| **前端-管理后台** | Vue 3 + Element Plus | 3.4 + 2.5 | 企业级标准 |
| **前端-小程序** | 微信原生 | - | 纯展示，简单即可 |
| **后端** | Spring Boot | 3.2.1 | Java企业级稳定 |
| **数据库** | SQL Server | 2022 | 客户指定 |
| **缓存** | Redis | 7.0 | 验证码、计数器 |
| **对象存储** | 阿里云OSS | - | 标书PDF、图片 |
| **地图** | ECharts + 高德地图 | 5.5.0 | 数据可视化 |

### 4.2 开发规范要点

**前端规范**：
- Nuxt.js必须使用`useFetch`进行SSR数据获取，不能用`axios + onMounted`
- 标书详情页必须做SEO优化（meta标签、结构化数据）
- 小程序所有支付按钮改为"请前往PC端支付"

**后端规范**：
- 所有合伙人查询必须加`WHERE binding_partner_id = ?`
- 标书编号生成必须加数据库锁，防止并发重复
- 福利分配必须事务保护，失败全回滚

**数据库规范**：
- 时间字段统一用`DATETIME`，不用`DATE`
- 分页必须用`OFFSET...FETCH NEXT`语法

### 9.1 常见问题与解决方案

**问题1：标书编号重复**
```sql
原因：高并发下，两个管理员同时发布标书
解决方案：
ALTER TABLE tender_sequences ADD CONSTRAINT UK_org_type UNIQUE (org_type);
-- 发布时先更新序列表再获取值
UPDATE tender_sequences SET current_value=current_value+1 WHERE org_type='A';
SELECT current_value FROM tender_sequences WHERE org_type='A';
```

**问题2：合伙人数据泄露**
```java
原因：SQL查询忘记加WHERE条件
解决方案：使用MyBatis拦截器强制注入
@Intercepts({@Signature(type = Executor.class, method = "query", ...)})
public class DataScopeInterceptor implements Interceptor {
    // 检测SQL中是否包含binding_partner_id过滤条件
    // 如果没有且当前登录人是合伙人，自动拼接WHERE条件
}
```

**问题3：支付回调失败**

原因：服务器防火墙拦截微信回调IP

1. 在服务器防火墙白名单添加微信支付回调IP段

### 10.1 技术关键点（不可妥协）

   - 验收方式：查看网页源代码，必须能看到完整HTML

   - 理由：这是商业模式的生死线，泄露会导致合伙人纠纷
   - 验收方式：渗透测试，尝试跨城市访问数据

3. **标书下载防重复扣费必须严格测试**
   - 理由：错扣会员次数会导致投诉和退费
   - 验收方式：并发测试，同一会员10个标签页同时下载

4. **支付二维码必须一单一码**
   - 理由：防止支付混乱和对账困难
   - 验收方式：生成10个订单，检查二维码URL是否唯一
